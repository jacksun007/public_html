Assignment 3: Object-Relational Mapping

Due Date: November 17th, 11:59pm

Lab TA: Jeffrey Fang

In this assignment, you will develop an object relational mapping (ORM) framework for EasyDB™ in Python.

Background

Object-relational mapping is a technique for converting database records to and from in-memory objects. Therefore, instead of writing code that extract values from a database query such as this:

# returns a Record object
rec = db.execute("GET 2 FROM User")
if rec.valid():
    name = rec["firstName"]

into this:

# returns a User object
try:
    user = User.get(db, id=2)
    name = user.firstName
except ObjectDoesNotExist:
    pass

The main advantage of an ORM layer is that it abstracts away the underlying database implementation, allowing you possible freedom to switch your database solution to another (e.g. from EasyDB to MySQL 😢).

Another important advantage is that ORM can help you convert to and from in-memory types to storage types seamlessly. For example, some relational databases, such as EasyDB, do not natively support an IP Address type. Therefore, to store an IP address in these databases, it must first be converted to a string (e.g. "127.0.0.1"). Unmarshalling code is required to read it back from the database and convert it back to an IP Address object. These tedious tasks can be automated with ORM.

EasyDB

EasyDB is a toy relational database that provides minimalistic features for managing structured data. In a relational database, data is organized into tables of columns and rows. A column in a database table is a set of data of a primitive type (that is supported by the database). A table is defined with a list of one or more named columns before any data can be inserted. A row is a set of related data that is given meaning through the table it exists in. To insert a row into a table, each value of the row must have the same type as what is defined in the table, in the same order. Also, the number of values in a row must be exactly equal to the number of columns defined in the table. For example, a User table may have the columns: firstName, lastName, height and age. A row in a User table may be ("Jay", "Sung", 5.5, 31), which correctly represents a single user in this table. A table, of course, can have many rows, allowing you to store multiple users in a User table.

Database Schema

There can also be multiple tables in a relational database, allowing tables to reference others. To specify the database schema (i.e. list of table definitions) in EasyDB, use the following syntax:

table-name {
    column-name-1 : type | table-specifier;
    column-name-2 : type | table-specifier;
    ...
}

In EasyDB, each column in a table can be one of four types: integer, float, string, or foreign key (i.e. a "pointer" to a record in another table). To specify a foreign key type, use the name of the referenced table. The names of a table or column must start with a letter, followed by letters (either case), numbers, or underscore. However, only the ORM layer may use the underscore character for column names (see Custom Type section for more information). In addition, the name of a column cannot be id or pk (explained below). Here is an example of a complete database schema:

User {
  firstName: string;
  lastName: string;
  height: float;
  age: integer;
}

Account {
  user: User;
  type: string;
  balance: float;
}

In the definitions above, we have two tables: User and Account. For every user, we track his/her names, height, and age. For an account, we store the account type and its balance. But most importantly, an account references the User table. user stores the id of a row in the User table. Every row in EasyDB has an automatically generated unique integer id value which allows the row to be unambiguously referenced. In this manner, a user can have zero or more associated accounts. Because the ORM layer references object id using either the name id or pk, we disallow those two names to be used for column names.

EasyDB currently disallows cycles in foreign keys. Here are two examples where a cycle occurs:

# not allowed in EasyDB
User {
  firstName: string;
  lastName: string;
  father: User;
  mother: User;
}

# also not allowed in EasyDB
User {
  firstName: string;
  lastName: string;
  pet: Animal;
}

Animal {
  species: string;
  owner: User;
}

Hint: you can guarantee that there's no cycle if a foreign field can only reference already defined table (except for the current one). This is how the instructor's version of the server works.

Access Protocol

To access a running EasyDB server, you must know the host name and its port number. Once you are connected, you can make one of five EasyDB queries that is available. Note that EasyDB assumes that the client (the program that is connected to the server) is aware of the current schema and does not provide an API for introspection. EasyDB uses binary format as its mean of communication. Therefore, we describe common structures in the network packets that are used in EasyDB. All structures are defined using C syntax and assumed to be 4-byte aligned, with int being 32-bit signed integer, and long being 64-bit signed integer. All integers are in network byte order.

enum Command {
    INSERT = 1,
    UPDATE = 2,
    DROP = 3,
    GET = 4,    
    SCAN = 5,
    EXIT = 6,
};

struct request {
    int command;  /* one of Command */
    int table;    /* table number */
};

enum Code {
    OK = 1,
    NOT_FOUND = 2,       /* id not found */
    BAD_TABLE = 3,       /* table not found */
    BAD_QUERY = 4,       /* error during scan */ 
    TXN_ABORT = 5,       /* update aborted */
    BAD_VALUE = 6,       /* column value type mismatch */
    BAD_ROW = 7,         /* number of values is incorrect */
    BAD_REQUEST = 8,     /* malformed packet */
    BAD_FOREIGN = 9,     /* foreign key not found */
    UNIMPLEMENTED = 10,  /* you can ignore this (you will return this for asst4) */
};

struct response {
    int code;      /* one of Code */
};

All EasyDB requests begin with request. The command field is one of the five commands listed above, and the table field is the table number. A table's number is the order in which it appears in the schema, starting at 1 (in earlier example, User's table number would be 1, and Account's would be 2). All responses begin with response, which contains a code for success (i.e. OK), or error. BAD_TABLE, BAD_ROW, BAD_VALUE, and BAD_REQUEST are possible for all EasyDB commands except for the Exit command (only BAD_REQUEST is possible).

enum Column {
    INTEGER = 1,
    FLOAT = 2,
    STRING = 3,
    FOREIGN = 4,
};

struct value {
    int type;     /* one of Column */
    int size;     /* size of buf */
    char buf[];   /* data stored within */
};

struct row {
    int count;    /* number of elements in values */
    struct value values[];
};

In some packets, we need to send a generic value, or a row of values. In these packets, struct value and struct row would be used. struct value contains the column type (type), the size of the buffer that is needed to store the value (size), and the buffer itself (buf). For all column types, size is equal to 8 (i.e. 8 bytes) except for string, which is stored in multiples of 4 bytes. Therefore, the size of the buffer for "bad" would be 4 bytes (3 bytes rounded up to 4), but 8 bytes for "hello" (5 bytes rounded up to 8). For struct row, count specifies how many values are stored.

Atomic Update
struct key {
    long id;      /* id of the row */
    long version; /* version of the row */
};

EasyDB supports a primitive version of atomic update. In a concurrent environment, atomic update ensures that changes made to a row will not conflict with changes made by others. In EasyDB, updating a piece of data requires retrieving the initial data followed by modifying it in some ways. For example, to withdraw $100 from an account, one needs to first know the current balance, and then deduct $100 from it. However, there is a window of time between two EasyDB commands where it is possible that someone else makes some changes to the same row. For example, the following problem may occur:

      You            Your Mom     EasyDB
  GET balance ---------------------->
                   GET balance ----->
                       1000    <-----
     1000     <----------------------
                  (deposit $500)
                SET balance 1500 --->
(withdraw $100)            
SET balance 900 -------------------->

As you can see above, your final balance is supposed to be $1400. However, due to problematic interleaving of the commands, the deposit made by your mother was lost. EasyDB protects from this situation by having a version number associated with every row. When you make an update, you pass the version number of the row that you were modifying to the database. If the version number mismatches with what is in the database, a conflict is detected and TXN_ABORT is returned, notifying you that your update was aborted, as seen in the figure below.

      You            Your Mom       EasyDB
  GET balance ---------------------->
                   GET balance ----->
                       1000  <------- v1
     1000   <------------------------ v1
                  (deposit $500)
                SET balance 1500 ---> v2
(withdraw $100)            
SET balance 900 --------------------> NO
   TXN_ABORT  <----------------------

As the programmer, it is likely that you would put an atomic update in a loop so that you can retry if abort occurs.

We will now describe the format of each EasyDB query.

Insert Command
struct requeststruct row

The insert command adds a new row to a database table. Its request packet starts with a request structure, followed by a row structure, which contains one or more rows. For the User table example, the following is an illustration of the packet that is sent to the server:

command: INSERT
table: 1		 count: 4 type: string
size: 4
buf: "Joe"		 type: string
size: 4
buf: "Sung"		 type: float
size: 8
buf: 5.5		 type: integer
size: 8
buf: 31

Next, the response packet is formatted as shown below:

struct responsestruct key

Note that struct key only appears if the response code is OK. The insert command can fail if foreign key reference is not found (BAD_FOREIGN). A foreign field is a 64-bit integer that refers to the id of a row in another table. If this field is set to zero, then it does not reference anything and it would not be possible to receive BAD_FOREIGN.

Update Command

The update command is used to update an existing row in a table. It is similar to insert, with the addition that struct key of the row needs to be specified. If you don't care about atomic update, set version of struct key to 0.

struct requeststruct keystruct row
struct responselong version

The response for the update command is similar to insert, except only the new version number is returned. NOT_FOUND is returned if the row is not found. TXN_ABORT is returned if atomic update failed. BAD_FOREIGN is returned if a foreign field reference does not exist.

Drop command

The drop command deletes a row in table. Therefore, it needs to specify the id of the row to be dropped. In EasyDB, the drop command performs cascade delete, which means that if a row is referenced by rows in other tables, those rows will also be deleted. For example, if a user has three associated accounts, upon deleting the user, all associated accounts will be deleted as well. This design ensures that there will be no dangling pointers in the database.

struct requestlong id

The response packet for drop is just struct response. NOT_FOUND is returned if the row is not found.

Get Command

The get command retrieves a row from a table, which requires specifying the id of the row.

struct requestlong id

The response packet, on success, will include the row and its associated version number. NOT_FOUND is returned if the row is not found.

struct requestlong versionstruct row
Scan Command
enum Operator {
    OP_EQ = 1,  /* equal */
    OP_NE = 2,  /* not equal */
    OP_LT = 3,  /* less than */
    OP_GT = 4,  /* greater than */
};
struct requestint column
int operator
struct value

The scan command compares a column of every row in a table against a value and returns the id of all rows that matched. The column field is the column number to be matched, based on the order it appears in the table definition, starting with 1. Column 0 corresponds to the internal id field. The column type of the id field is FOREIGN The operator field specifies how to compare against the value stored in struct value.

struct responseint count
long ids[]

On success, the response will contain zero or more ids (zero is possible if there were no matches). BAD_QUERY is returned if one of the following errors occurred: 1) column type and value type mismatched, or 2) operator for the given type is not supported. In EasyDB, id and foreign field only support OP_EQ and OP_NE. For string, lexicographical comparison is performed (e.g., "A" is less than "B" and "5" is less than "-2").

Exit Command
struct request

The exit command shuts down the connection gracefully. There is no server response for this command. The server will disconnect once it receives this command. You may set any value for the table field, but 1 is preferred.

Solution Requirements

This assignment has a couple of milestones, each of which can work independently and thus are worth partial marks. You and your partner should take on one of each part of the assignment.

Database Client

In order to communicate with the database server, we need to write a client library that will send requests to an EasyDB server as well as parsing responses received from the server. We will describe the interface here.

Database Schema

As said earlier, the server assumes the client knows the schema, so we must supply the schema to the client to instantiate it. Here is how an ORM developer would use your EasyDB library:

import easydb

tb = (
    ("User", (                  # table_name
        ("firstName", str),     # (column_name, type)
        ("lastName", str),
        ("height", float),
        ("age", int),
    )),
    
    ("Account", (
        ("user", "User"),       # (column_name, table_reference)
        ("type", str),
        ("balance", float),
    ))
)

db = easydb.Database(tb)

The above code creates a Database object. If there is an issue with the provided schema, then it should throw the following errors:

  • TypeError: thrown when either table name or column name is not a string.
  • ValueError: thrown when type is not one of str, float, int, or a string referencing another table name.
  • orm.IntegrityError: thrown when foreign keys causes a cycle or if foreign field references a nonexistent table.

Next, we connect to the server via the connect member function.

db.connect("127.0.0.1", 8080)

connect takes two arguments, the host name and the port number. You should use Python's socket library to implement this function. The function should throw whatever error socket.connect throws (Hint: don't error check on any of socket's methods).

Next, implement the close method to disconnect from the server and clean up the socket. You should send the Exit command before closing the connection to inform the server that you are shutting down the connection gracefully.

db.close()

The main part of this milestone is implementing all of EasyDB commands. You are suggested to use Python's struct library to help build packets and read responses from the server. When the server responds in error, you should throw objects of the corresponding error classes.

  • orm.ObjectDoesNotExist: corresponds to the error code NOT_FOUND.
  • orm.InvalidReference: corresponds to the error code BAD_FOREIGN.
  • orm.TransactionAbort: corresponds to the error code TXN_ABORT.
  • orm.PacketError: corresponds to the error code BAD_TABLE, BAD_VALUE, BAD_ROW, and BAD_REQUEST. They all fall under the same category because either the ORM layer or the database client is supposed to have caught these mistakes before sending off the request.

Now, here is the specification for each command:

Insert Command
insert(table_name, values)

table_name is the name of the table, and values should be a sequence (e.g., list, tuple) of N elements where N is the number of columns in the table. You should throw a ValueError if any of the arguments have problems, such as the table name does not exist, the number of elements in values is incorrect, or one of the columns have an invalid type (i.e., must be one of the native types that EasyDB supports). If the response is OK, return a tuple of pk, version, where pk is the id only the newly created object. For example:

pk, version = db.insert("User", ["Jay", "Sung", 5.5, 31])
Update Command
insert(table_name, pk, values, version=0)

Same as insert, with two more arguments, pk and version. pk is the id of the row to update, and version is optional, and if specified, will enable atomic update. If the response is OK, return the new version, regardless of whether atomic update was enabled or not.

Drop Command
drop(table_name, pk)

table_name is the name of the table, and pk is the id of the row to be deleted. This method does not return a value, but orm.ObjectDoesNotExist is raised if the id does not exist in the specified table.

Get Command
get(table_name, pk)

table_name is the name of the table, and pk is the id of the row to be retrieved. If the response is OK, return a list of the values and the version. For example:

values, version = db.insert("User", 1)
print(values)
# ['Jay', 'Sung', 5.5, 31]
Scan Command
scan(table_name, query)

table_name is the name of the table, and query is a tuple of 3 elements: name of column, the operator, and the right operand, in that order. Similar to insert, ValueError should be thrown if the column name does not exist, operator is not supported, or the right operand is the wrong data type. If the response is OK, return a list of all matching ids (may be empty). For example:

import orm.easydb
results = db.scan("Account", ("balance", orm.easydb.OP_GT, 10000))
print(results)
# [1, 3, 7]

Database Model

In an object-relational mapping software, the programmer would write their database schema in a database-neutral language (i.e., in a way that the underlying database could be swapped out without affecting the program). For our ORM, the programmer will write the schema in Python (inspired by Django). Let's first look at an example schema:

# schema.py

import orm

class User(orm.Table):
    firstName = orm.String()
    lastName = orm.String()
    height = orm.Float(blank=True)
    age = orm.Integer(blank=True)
    
    def __repr__(self):
      return "<User: %s %s>"%(self.firstName, self.lastName)
    
class Account(orm.Table):
    user = orm.Foreign(User)
    type = orm.String(choices=["Savings", "Chequing",], default="Chequing")
    balance = orm.Float()

For all database tables, the client will inherit orm.Table. Column names must start with a letter, followed by letters of either case, or numbers, but underscore is not allowed (the ORM layer uses it for special purposes). For each column in the table, it is presented by an instance of one of the following classes: String, Integer, Float, and Foreign. Each of these classes corresponds to a primitive type that EasyDB supports. For the first three classes, there are no positional parameters. For Foreign, it takes one positional parameter: the class of the referenced table. For later part of this assignment, you will be adding new field types and building them on top of the primitive types.

Every field types, except for Foreign, support the following options:

  • blank: this field is allowed to be left unspecified when creating new objects. When this option is set to False (which is the default), AttributeError will be raised when attempting to create a new object if the field is not specified. Otherwise, default value of the type (e.g. 0 for int, "" for string) will automatically be used for objects with the field missing.
  • default: the default value for this field if it is unspecified in the constructor. Unlike blank, default will automatically set the missing attribute to the default value. .
  • choices: a list of allowed values for this field. ValueError will be raised if the field has value not in choices when insert or update is attempted.

Foreign, on the other hand, only supports blank. If blank is true, the field can reference nothing (i.e. set to None).

Our ORM provides two interfaces to initialize the underlying database client.

Setup
orm.setup(database_name, module)

setup will return a database object that is initialized, but not yet connected. database_name is the name of the underlying database implementation. Currently, we only support one argument: "easydb". module is the module that contains the schema (e.g. schema.py). 

import schema
db = orm.setup("easydb", schema)  # this should create easydb.Database
db.connect("127.0.0.1", 8080)
Export
orm.export(database_name, module)

export will return a string which can be read by the underlying database to create the corresponding database tables, each with the specified columns, similar to the example shown in the Database Schema section. This feature is helpful so that you do not have to write the database schema manually in EasyDB format on top of writing the Python ORM format. For example: the following Python schema:

class Course(orm.Table):
    instructor = orm.Foreign(Person)
    name = orm.String()
    difficulty = orm.Float(default=5.0)
    numEnrolled = orm.Integer(blank=True)   # cannot use underscore (num_enrolled not allowed)

is exported as a string shown here:

Course {
    instructor : Person;
    name : string;
    difficulty : float;
    numEnrolled : integer;
}

Note that if there are multiple tables in the schema, then there will be multiple tables in the exported format. Also, all field options are ignored because they are only enforced by the ORM layer. The user of the export function will likely save the output to a file to be used to initiate the database server.

Table Methods

For each table in the schema, the following class methods are defined. For each of these methods, Table should be replaced by the actual Table in the schema (e.g., User, Account).

Object Interface
object = Table(db, ...)

Create a new object (row) of type Table. The constructor accepts keyword arguments which are the column names of the table, and one positional argument: the database object. Default values are used for unspecified columns. If blank is not true and no default is specified, then the field must be initialized in the constructor. For example:

# joe.height is 0.0
joe = User(db, firstName="Joe", lastName="Harris", age=32)

Each instance has the following attributes on top of the column fields:

  • save: Saves the object to the database. For new objects in EasyDB, this will call insert. For existing objects, this will call update. For fields where blank is true, the field is set to the default type value. save takes an optional parameter, atomic, which is by default true. If set to false, disables atomic update for that save.
  • delete: Deletes the object. In EasyDB, this will call the drop command.
  • pk: the id of the object (i.e. primary key). This field is None before it is in the database.
  • version: the version of the object. This field is None before it is in the database.

Here is an example of an object's usage, continuing from the above example:

joe.pk is None    # True
joe.height == 0.  # True
joe.lastName = "Smith"
joe.save()
joe.pk            # 2 (the id of the object)
joe.version       # 1 (the version of the object)
joe.age = 33
joe.save(atomic=False)    # disable atomic update for this call to easydb.update
joe.version               # 2 (new version of the object)
joe.age = 34
joe.save()                # this will call easydb.update, which may raise TransactionAbort
joe.delete()              # deletes the row from the database
joe.pk is None            # True
joe.save()                #
joe.pk                    # 4 (the new id of the object)

Note that when setting field value, if the type does not match, TypeError should be thrown. The only exception is when assigning an integer to a float field. In this case, you should accept implicit type conversion. For example:

joe.age = "55"    # NO - assigning string to integer field
joe.height = 3    # OK - assigning integer to float field
joe.age = 3.5     # NO - assigning float to integer field

All foreign key references are converted automatically to their objects. Hint: you can implement this lazily (only retrieve from database when field is accessed), or eagerly (retrieve foreign keys during the get ORM operation). You should raise InvalidReference if the referenced object is not in the database (you will know this when the server returns BAD_FOREIGN).

account = Account.get(db, pk=2)
account.user  # <User: Joe Smith>

When calling save(), if referenced object is already in the database (i.e. pk field is set and not None), then you should only save the current object. However, if it is not in the database when you save another object, the referenced object will be saved first before the object is saved. For example:

greg = User(db, firstName="Greg", lastName="Russell", age=27)
account = Account(db, user=greg, type="Chequing", balance=100.0)
account.save()  # will save 'greg' first before saving 'account'
saving = Account(db, user=greg, type="Saving", balance=200.0)
saving.save()   # will not save 'greg' first since it already exists in database

The above scenario can be recursive for multiple levels of references. Note that for a foreign field, you must set it with an object of the expected type, or None is the field is allowed to be blank.

account.user = greg             # OK
account.user = 3                # NO - cannot use object id
account.user = User.get(db, 3)  # OK
account.user = None             # NO - blank is not specified for this field
account.user = account2         # NO - incorrect object type (only instance of User accepted)

You are suggested to use descriptors to implement fields. However, other implementations are possible, such as using __getattr__, __getattribute__, or property. You may not defer type checking until save(), so using an ordinary data field is not a viable option.

Get Method
object = Table.get(db, pk)

Returns an existing object from the table, if it exists. The returned object has the same interface as above, with pk and version already set.

Filter Method
[object...] = Table.filter(db, ...)

Returns a list of objects that matches the query. If no argument is given, returns all objects in the table. The query arguments uses the following format: columnname__op=value where columnname is the column name of the object, op is the operator, and value is the right operand. For the equality operator, __eq is not needed. For example:

results = User.filter(db, age__gt=33)
...
results = User.filter(db, firstName="Joe")
results # [<User: Joe Smith>]

The supported operators are ne (not equal), gt (greater than), and lt (less than). When the field is a foreign key, you can use either the object itself or its id. For example:

results = Account.filter(db, user=joe)
# or
results = Account.filter(db, user=2)
Count Method
number = Table.count(db, ...)

Returns the number of matches given the query. If no argument is given, returns the number of rows in the table. The query format is the same as above. You can use this method to check if an object exists without fetching the entire object.

if Table.count(db, pk=2) == 1:
    # id exists
else:
    # id does not exist

Note that object id only supports equality and inequality and no other operators.

Custom Types

In this section, you will build custom field types that is not natively supported by the underlying database. Your ORM layer will seamlessly convert to and from the database's storage type, and perform type safety checks as is required.

Datetime Field

Create a new field, DateTime, which is represented in Python by a datetime.datetime instance. For example:

# schema.py
import orm
from datetime import datetime

class Lecture(orm.Table):
    location = orm.String()
    start = orm.DateTime(default=datetime.now)
    end = orm.DateTime()

# main.py
ece326 = Lecture(db, location="BA1190", end=datetime(2019, 10, 15, 13))
ece326.start = datetime(2019, 10, 15, 12)
ece326.save()

For the default value, you must support calling a function if default is assigned a function. For the datetime instance, you can ignore time zone related settings.

Hint: store it in POSIX timestamp format.

Coordinate Field

Create a new field, Coordinate, which is represented in Python by a 2-tuple, and must be a valid geographic coordinate in latitude and longitude. For example:

      
class Capital(orm.Table):
    name = orm.String()
    location = orm.Coordinate()      

# main.py
ottawa = Capital.filter(db, name="Ottawa")[0]
ottawa.location   #(45.4215, 75.6972)

If the coordinate is invalid, you must raise ValueError when save or update is attempted. You only need to support equality and inequality operator when this field is queried. You may assume that you will always read back a valid coordinate from the database.

Hint: store it using two native EasyDB fields, and use the underscore character so that there is no conflict with user-defined column names. For example:

Capital {
    name : string;
    location_lat : float;
    location_lon : float;
}

Testing Your Code

To test your database client, you will need to run the server. You can start the server by running easydb -g PORT on one terminal while running the client on a different terminal, where the -g option will print everything the database is doing, and PORT is the port number. You should choose a unique port number so that other people also working on your machine would not accidentally connect to your database. By default, the server will load default.txt for the database schema, but you will need to supply a different one if your ORM schema differs. Type easydb to see the usage on how to specify a different schema file.

To test the database client or the ORM layer, you can either write a custom script that imports schema.py, or you can run in interactive mode using the main.py run command. It will start interactive mode with all the classes and functions defined in schema.py loaded, as well as automatically connecting to the database, and storing the database object as db. For example:

student:~/asst3$ ./main.py run
Python 3.5.2 (default, Oct  8 2019, 13:06:37) 
[GCC 5.4.0 20160609] on linux
Type "help", "copyright", "credits" or "license" for more information.
(InteractiveConsole)
>>> db
<EasyDB Database object>
>>> user = User(db, firstName="Greg", lastName="Russell", age=27)
>>> user.save()
>>> db.get("User", 1)
['Greg', 'Russell', 0.0, 27]
>>>
