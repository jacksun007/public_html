Assignment Guideline

All assignments will be done in groups of 2 students. You should sign up to a group as soon as possible on The Quercus website. Please make use of Piazza to facilitate finding a partner.

You should be aware that this course will require significant time for each assignment. Many of these assignments require significant preparation that involves reading the specification and the code provided for the assignment. Please read the handouts carefully before starting. This preparation should be done before the lab sessions otherwise you will not be able to make the best use out of the lab TA's time. Preferably, you should have already made some progress so that you can use the lab hours to test your code and, if needed, get help from the lab TAs. Therefore, it is very important to start as early as possible on your assignments and avoid procrastination.

It is your responsibility to correctly identify and meet the deadline for each assignment. Assignments will not be accepted after their due dates, without prior arrangement with the course instructor. In general, no deadline extensions will be granted to individuals except in severe circumstances.

Assignments will be marked by the TAs after they are submitted (they are not marked in the labs). The TAs will send you an email with your lab marks. Note that attending the lab is optional. You can do the assignments at home and submit them remotely. However, it is important that you get your code to work correctly on the lab machines, no matter where you develop it. A program that does not work correctly on the lab machines, even if it works correctly on your home machine, will be marked as incorrect. Plan ahead, and give yourself the time to test what you developed at home on the lab machines before the deadline.

In general, the TAs will be on duty only during the scheduled lab session. To get help at other times, you should use the forum on the Piazza website. The TAs and the instructors will regularly check the forum and answer your questions. These answers become available to all students. Thus, please check the forum before posting a question to make sure that the question has not been posted and answered earlier. The forum provides functionality for you to receive emails as any messages or their answers are posted. Please enable this functionality if you want to know when your questions have been answered (or when any new questions are posed).

You are encouraged to use the forum, and to engage in discussions about the assignments with fellow students. Please do not post code publicly on the forum. Doing so will be treated as an academic offense. You can also post messages on Piazza that are only visible to TAs and the instructor.

Lab Information

We will be using UG Linux machines for grading the labs. Although most or all of this lab should "just work" in many other environments (Cygwin, Solaris, etc.), the course staff will not be able to assist in setting up or debugging problems caused by differences in the environment. If you choose to do development in an unsupported environment, it is your responsibility to leave adequate time to port your solution to the supported environment, test it there, and fix any problems that manifest.

All source code must be compiled and submitted on the lab computers. These computers are Linux machines with names that matches ug*.eecg.utoronto.ca, e.g., ug149.eecg.utoronto.ca. You can tell which machine you are using from the command prompt. For example, if you are using ug149.eecg.utoronto.ca, the command prompt will show ug149. The range of machines that are accessible is roughly ug51-ug100, ug132-ug180 and ug201-250.

Each group will be assigned a group number for this course. You can find out your group number by typing the groups command after logging into the lab machines.

$ groups

This command provides information about all the Unix groups to which you belong. One of them should be in the range from pl-005 to pl-100. The 3-digit number after pl- is your group number for this course (e.g., with 031, the group number is 031 and not 31). If you do not have a group number, please contact one of the TAs and/or make a private post on Piazza about the issue. Make sure you have already signed up on Quercus prior to notifying the instructor or the TAs.

Basic Setup

On your first log-in, you should change the password on your account for security purposes. You can use the passwd command to do so.

Start by setting up your personal information for Git, if you have not done it previously. The "--global" option will update the .gitconfig file in your home directory.

cd
git config --global user.name "Your Name"
git config --global user.email you@example.com

Git is a version control software for tracking changes to your source code. You and your partner will use Git to help manage your code base. Git is also used for you to submit your assignment, thus it is important for you to learn to use Git correctly and effectively.

Next step, make sure that your home directory is not accessible to others. This way you can ensure that you code will not be available to others accidentally.

cd
chmod 700 .

At this point, one of the group members should follow the instruction under Remote Repository, while the other group member should follow the instruction under Cloning Repository after the remote repository has been created.

Remote Repository

You will need to do this step once before starting assignment 1 and only one group member should perform this step.

Create a new ece326 directory and initialize your Git repository in that directory.

cd
mkdir ece326
cd ece326
git init

The last command should show the following:

	Initialized empty Git repository in ../ece326/.git/

At this point, we (i.e., the instructor and the TAs) cannot access this repository because it is local within your home directory (that should no be accessible to others). Instead, you will submit your assignment code by creating a clone (copy) of your repository to a remote repository location that is accessible to you and to us.

You need to run these instructions once to setup your remote repository. After that, you can update the remote repository by following the instructions in the next step.

Each of you has been assigned a group number for this course. Below, we use XXX to denote your 3-digit group number. The remote repository will be located on the ug250.eecg.utoronto.ca machine. Log in this machine.

ssh ug250.eecg.utoronto.ca

Now create your repository as follows:

cd /srv/ece326f/pl-XXX
mkdir ece326
chmod 770 ece326
cd ece326
git init --bare --shared=group

Make sure to replace XXX with your 3-digit (not 2-digit) group number. The chmod command above ensures that the repository will be accessible to the instructor and the TAs but not to other students. The git init command creates an empty Git repository.

Now, you are done with this machine, so exit out of it. Back on your local machine, you need to tell the local repository about the location of the remote repository.

cd ~/ece326
git remote add origin ssh://ug250.eecg.toronto.edu/srv/ece326f/pl-XXX/ece326

The command shown above is run in your local repository. It associates the name origin with the remote repository. You can see that by using the git remote command. 

git remote -v

It will show the following:

    origin  ssh://ug250.eecg.toronto.edu/srv/ece326f/pl-XXX/ece326 (fetch)
    origin  ssh://ug250.eecg.toronto.edu/srv/ece326f/pl-XXX/ece326 (push)
Ignoring Temporary Files

Now you are ready to update your remote repository. Let's try to commit a file and make it available in the remote repository. .gitignore is a special file that Git reads to exclude certain files from showing up as candidates to be added to the repository. It is common to place temporary file types (e.g., .o files created during compilation, or .log files created by the tester) in .gitignore so we do not bloat our repository unnecessarily. Create a .gitignore file in your ece326 folder:

cd ~/ece326
nano .gitignore

Your terminal should now enter into a text editor. Copy paste the following:

*.o
*.pyc
*.log
*.mk

The above text means ignore any file that ends with .o, .pyc, .log, or .mk. Now save and quit from the editor by pressing Ctrl + X (Exit) followed by Y (Yes, Save Changes). Now, type git status in the terminal and you should see the following:

Untracked files:
  (use "git add <file>..." to include in what will be committed)

        .gitignore

no changes added to commit (use "git add" and/or "git commit -a")

Your next step is to stage the file for commit, which means to add a file into the list of files to be committed. You can do that by entering git add .gitignore. Now, type git status to see what happens:

Changes to be committed:
  (use "git reset HEAD <file>..." to unstage)

        new file:   .gitignore

The new file we created is ready to be committed, which means the repository will track the file and possibly keep multiple versions of it for your debugging or auditing purposes. To commit the all staged files, type the following:

git commit -m "added .gitignore"

The "-m" option allows you to associate a message about what changed to the commit. You should see a message that says 1 file changed, 1 insertion. Now we want to push the changes that we made to the remote repository, you can do so with the following command:

git push origin master

You should see something that ends with:

To ssh://ug250.eecg.toronto.edu/srv/ece326f/pl-XXX/ece326
   xxxxxxx..yyyyyyy  master -> master

You're done! It is now time for your partner to get a copy of the repository.

Cloning Repository

Once the remote repository is set up, your partner (or you, if you accidentally deleted the entire ece326 folder), can make a copy of the repository by cloning it through this command:

cd ~
git clone ssh://ug250.eecg.toronto.edu/srv/ece326f/pl-XXX/ece326

This will create a local repository in ~/ece326. Now both you and your partner have a copy of the repository and are connected to the same remote repository.

Working Locally

Once you have made progress on your assignment, you should consider committing your work to the repository so that if you need to rollback to earlier versions of your work, it is saved in the repository. Also, your partner may want to continue where you left off, and thus you will need to commit your changes and make them available to your partner via the remote repository.

The first step to committing your changes is to see which files have been changed, or added. You can do so using the git status command in the current assignment's folder:

	Changes not staged for commit:
	  (use "git add <file>..." to update what will be committed)
	  (use "git checkout -- <file>..." to discard changes in working directory)
	
		modified:   foo.py
		modified:   bar.py
	
	Untracked files:
	  (use "git add <file>..." to include in what will be committed)
	
		baz.py
	
	no changes added to commit (use "git add" and/or "git commit -a")
The command shows the files that you have modified, and some untracked files. The tester.

If you only want to add specific files to the commit, you may do so like this:

git add foo.py baz.py

However, if you want to add everything changed or added in this folder, you can do:

git add .

Beware that the above command may add unwanted files as well. To ignore these unwanted files, add their full name or their extensions to the .gitignore file as explained earlier under Remote Repository.

Now, you can commit these changes via the commit command.

git commit -m "Updated foo and added baz"

Next, you can update your remote repository by running the git push command shown below. This command will only push the files that you have committed to your local repository, so make sure to commit all the relevant files to your local repository that you would like to push to the remote repository.

git push -u origin master --tags

When you run the command above, you will see:

    Branch master set up to track remote branch master from origin.

The "--tags" option pushes your local tags to the remote repository. We will be using these tags to mark your assignments, so don't forget to add the tags option. At the end of every assignment handout, we will specify the assignment submission tag that you need to use to submit your work.

If you run git status, you should see the following:

    On branch master
    Your branch is up-to-date with 'origin/master'.
    
    nothing to commit, working directory clean

You need to only run the push command with all the options shown above once. After that, you can push your commits in the local repository to the remote repository by typing the following commands in your local repository:

    git push
    git push --tags

When you add a new tag (e.g., Asst1-end) and run the push command shown above, it will show you:

    To ssh://ug250.eecg.utoronto.ca/srv/ece326f/pl-XXX/ece326
     * [new tag]         Asst1-end -> Asst1-end

You may re-submit your code any number of times. For example, you may modify your code and update the end tag in the local repository. To re-submit this updated code, you will need to force an update to the tag on the remote repository by using the "-f" option. Here is the full sequence of operations:

git tag -d Asst1-end // Deletes tag from previous submission for Assignment 1
git tag Asst1-end    // Creates tag for current submission for Assignment 1
git push --tags -f   // The "-f" option forces the tag to be pushed to the remote repository, even if the tag exist there

You can update the remote repository at any time as shown above. The instruction for submitting your work will be detailed in the handout for each assignment.

Working Together

After having made progress in an assignment, a group member will have committed meaningful work to their local repository. When the other partner begins to work, he/she will want to continue from these changes. For the partner to get these changes, the commits must be first pushed to the remote repository and then pulled to the partner's local repository.

The group member who has made the commits (e.g. member1) should push their commits to the remote repository.

member1@ug123: cd ~/ece326
member1@ug123: git push origin master

Finally the other group member (member2) can pull the commits (done by member1) to his/her local repository.

member2@ug123: cd ~/ece326
member2@ug123: git pull origin master

If conflicts arise, it means that you and your partner have made individual changes to a file and then committed your changes to the file separately. One of you will need to merge your code before you can continue. To avoid conflict, you and your partner should decide who will modify which files. If you need to modify your partner's files, ask him/her to commit his changes first, pull all of his changes from the remote repository, and make the changes. In the mean time, your partner should not continue working until you have committed and pushed all of your changes to his files.

Testing Your Code

We have provided you a program called ece326-tester that allows you to 1) test your code with various test cases, and 2) allows simulating the way we will be marking your code automatically. Please use this program before submitting your code so that you can catch obvious compilation errors, output errors, source code management errors, etc.

Tester Setup

You will need to setup your path to make it simpler for you to access the tester. To do so, you should check whether you use csh or bash as follows:

% echo $0

If the output shows sh or bash, then you are using bash. If the output shows csh or tcsh, then you are using csh.

  1. For csh, add the following to the end of your ~/.cshrc:

    set path=( /cad2/ece326f/tester/bin $path)

  2. For bash, add the following to the end of your ~/.bashrc:

    export PATH=/cad2/ece326f/tester/bin:$PATH

Then make sure to log out and log back in. Run echo $PATH and you should see the new path.

Running Tester

Please use the following commands after you have compiled and run your code. For example, for assignment 1, you should write the program described in the assignment 1 handout, and then run the tester.

% cd ~/ece326/asst1
% ece326-tester 1

You will need to be in the directory associated with the assignment (e.g., asst1 for assignment 1, asst2 for assignment 2, etc), and specify the assignment number for which the tests should be run. For example, for assignment 2, lab_nr is 2. For each test, you will see one or more PASS or FAIL results.

You can see the output that is expected by the tester by specifying the -v (verbose output) option to the tester. Note that the tester looks for regular expressions, not exact matches. Please use ece326-tester -h for more details.

The output of the tester will be stored in two files in the directory in which the tester is run:

  • tester.log: Shows the output of your program when running the ece326 tester.
  • tester.out: Summarizes the marks for the various tests.
Frequently Asked Questions

My output is correct but the tester says my program failed. Why?

Mostly likely, your output is incorrect in one of two ways:

  1. You have incorrect capitalization (e.g. "Hello World" instead of "Hello world").
  2. You added spurious spaces or indentation, (e.g. "Hello " with many spaces following the word is not the same as "Hello" to the tester)

What does a tester TIMEOUT mean?

The expected output is not found within the time limit.

The tester showed this, what does this error mean?

END OF FILE ERROR 
FAIL

The end-of-file error suggests that the tester was still looking for input but your program exited. Running the tester with -v will show you what input the tester is expecting. Generally, it means that your program has crashed or failed some assertion.

Simulating Marking

The tester helps you test your code. However, you need to simulate the marking functionality of the tester to find out whether you have submitted your code correctly. Before you do this step, make sure to submit your code, or else the marking functionality will not work correctly.

Then, you can simulate the marking functionality by adding the -m option to the ece326-tester command:

  % cd ~/ece326
  # make a brand new directory
  % mkdir marker
  % cd marker
  % ece326-tester -m lab_nr

The -m option will simulate the marking functionality. It will checkout your code from your remote repository (which you have created as part of submitting your code) using the assignment's end tag (e.g., Asst1-end), build the code, and then run the testing functionality on this code. Note lab_nr is the assignment number (1, 2, etc.).

The output will be stored in three files, shown below, in the marker directory where you ran the tester. The XXX shown below is your user number. The last two files are copies of the original outputs of the tester.

  • marker-XXX.log: Shows the output of retrieving files from your remote repository, building your code, and running the marker functionality.
  • tester-XXX.log: Shows the output of your program when running the ece326 tester.
  • tester-XXX.out: Summarizes the marks for the various tests.

We will run the same marker program to mark your assignment. You will receive these files by email when we finish marking your assignment. Let us know if you don't receive these files within one week after the assignment submission deadline.

We do not use any "hidden" test cases, so there are no surprises. Make sure to run the marker before submitting your code - it will tell you the marks you will get for your submission! If the tester and the marker is giving you different results for an assignment, it is possible that you have not tagged the latest version of your code since the marker uses git tags to checkout your code.
